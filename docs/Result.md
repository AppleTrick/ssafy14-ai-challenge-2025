### 2. 모델 업그레이드: Qwen2.5 → Qwen3

### 3. 병렬처리 vs 단일처리 성능 차이 발견 🔍

**실제 발견한 사실**:

| 처리 방식 | 정확도  | 비고      |
| --------- | ------- | --------- |
| 병렬처리  | 0.91358 | 약간 낮음 |
| 단일처리  | 0.91615 | 최고 성능 |

**⚠️ 주의: 원인은 명확하지 않습니다**

위의 차이는 실제로 관찰했지만, **정확한 원인을 실험적으로 확인하지 못했습니다**. 아래는 참고 자료를 바탕으로 한 **추론**입니다.

1. **Padding 설정 영향 가능성** (확정 아님)
   - 병렬처리 시 배치 내 샘플 길이 차이로 인한 padding 영향 가능
   - 하지만 이것이 주된 원인인지는 **확실하지 않음**
2. **토큰 출력 처리 차이** (추측)

   - 단일처리는 각 샘플을 독립적으로 처리
   - 병렬처리는 배치 내 상호작용으로 인한 미세한 성능 저하 가능성

3. **이론상 해결 방안** (실제 시도 못함):

   ```python
   # 이론상 권장되지만, 실제 효과 미확인
   processor.tokenizer.padding_side = 'left'

   # 이론상 필요하지만, 실제 효과 미확인
   generated_ids = out_ids[:, input_length:]
   ```

**교훈**: 병렬처리와 단일처리 사이에 성능 차이가 있었지만, **정확한 원인을 규명하지 못한 것이 가장 큰 아쉬움**입니다.

### 4. 학습 시간 문제 해결: 이미지 축소

**문제 상황**:

- Qwen3-VL-4B 모델 학습에 **6시간 이상** 소요
- 제한된 GPU 시간(16시간) 내에 반복 실험 어려움

**해결 방법**: 이미지 크기 축소

- 이미지 해상도를 줄여 메모리 사용량 감소
- 처리 속도 향상으로 학습 시간 단축

**트레이드오프**:

- ✅ 학습 시간 단축
- ❌ 이미지 정보 손실로 인한 성능 저하 가능성
- ❌ 최종 모델 성능에 부정적 영향

**교훈**: 시간 제약과 성능 사이의 균형이 중요하지만, 이미지 품질은 성능에 직접적 영향

### 5. 추론 시간 최적화 실패 ❌

**문제**:

- 학습된 모델로 추론 시 **2시간 소요**
- GPU 시간 부족으로 최적화 시도 제한

**시도했던 방법**:

1. **A100 GPU 40GB 활용**

   - 양자화 모델 사용으로 추가 최적화 여지 있음
   - 배치 크기 증가 가능성 존재

2. **배치 크기 조정 실패**
   - 추론 시 배치 크기를 더 높일 수 있는데 제대로 활용 못함
   - 시간 부족으로 실험 기회 제한

**실패 원인**:

- 제한된 시간 내 실험을 최우선으로 해야 했음
- 배치 크기 최적화보다는 모델 성능에 집중
- GPU 메모리 여유를 활용한 배치 크기 증가 실패

**개선 방안**:

```python
# 추론 시 배치 크기 대폭 증가 가능

# 추론은 학습 대비 메모리 사용량이 훨씬 적음

```

### 6. Fine-tuning 최적화 (중요도: ⭐⭐⭐)

**시도한 최적화 방법들**:

#### ✅ 데이터 로딩 최적화 (중요도: ⭐⭐⭐)

- 문제: NUM_WORKERS=0으로 GPU가 대기
- 해결: NUM_WORKERS=4, prefetch_factor=4, persistent_workers=True
- 효과: GPU 유휴 시간 제거

#### ✅ 이미지 전처리 간소화 (중요도: ⭐⭐⭐)

- 문제: 이중 다운스케일 작업
- 해결: 단일 리사이즈로 변경
- 효과: 전처리 시간 50% 단축

#### ✅ 데이터타입 통일 (중요도: ⭐⭐)

- 문제: fp16 ↔ bf16 캐스팅 오버헤드
- 해결: bfloat16으로 통일
- 효과: 캐스팅 오버헤드 제거

#### ✅ Gradient Checkpointing 비활성화 (중요도: ⭐⭐⭐)

- 문제: 메모리 절약 대비 속도 저하
- 해결: 메모리 충분 시 비활성화
- 효과: 1.5~2배 속도 향상

#### ✅ Scheduler Step 수 수정 (중요도: ⭐)

- 문제: GRAD_ACCUM 미반영
- 해결: update_steps_per_epoch 계산 반영
- 효과: 정확한 warmup/decay 스케줄링

**최종 설정**:

```python
BATCH_SIZE = 4
GRAD_ACCUM = 4
IMAGE_SIZE = 224
NUM_WORKERS = 4
batch 처리 속도: 13초 → 3~4초
```

## 💡 핵심 교훈 및 시사점

### ⚠️ 주의사항

본 챌린지에서 제한적인 시간 내에 복잡한 모델 최적화를 진행했으며, 코드를 전부 이해하지 못한 부분이 있습니다. 일부 최적화 이론이나 코드는 참고 자료를 바탕으로 예상치를 작성한 부분이 있습니다.

### 1. Padding Side의 복잡성 (추측)

**가설**: 병렬처리 시 배치 내 샘플 길이 차이로 인한 padding이 성능에 영향을 줄 수 있음

- Decoder-only 모델에서 **left padding**이 권장됨
- 하지만 이것이 실제로 점수 하락의 주된 원인인지는 **확실하지 않음**
- padding_side 설정은 많은 요인 중 하나일 수 있음

```python
# 이론상 권장되지만, 실제 효과는 실험 필요
processor.tokenizer.padding_side = 'left'  # 장단점 존재
```

**추론**: 병렬처리가 0.91358, 단일처리가 0.91615로 차이를 보였으나, 이 차이가 정확히 padding 때문인지는 실험적으로 확인하지 못했습니다.

### 2. 출력에서 입력 제거 (이론상 필요)

배치 처리 시 생성된 토큰만 추출하는 것은 이론적으로 필요한 과정

```python
generated_ids = out_ids[:, input_length:]
```

### 3. 데이터 로딩 최적화의 중요성 (이론)

- NUM_WORKERS=0은 GPU를 놀게 만듦
- 특히 VLM(Vision-Language Model)에서는 멀티프로세싱이 필수
- **실제로 적용은 제한적으로만 시도**

### 4. 전처리 파이프라인 최적화 (이론상)

- 불필요한 이중 리사이징 제거
- 고품질 필터보다 빠른 필터 고려
- **이미지 축소를 시도했으나 성능 하락 초래**

### 5. Gradient Checkpointing 트레이드오프 (참고 자료 기반)

- 메모리 절약 vs 속도 저하
- 메모리 충분하면 OFF가 성능상 유리
- **참고 자료 기반 정보**

### 6. 추론과 학습의 차이 (이론)

- 학습: 그래디언트, 옵티마이저 상태로 메모리 많이 사용
- 추론: 모델 가중치만 사용 → 큰 배치 가능 (3-8배)
- **실제로는 추론 최적화에 실패**

### 7. 배치 크기 최대화 (이론상)

- 작은 배치 크기는 GPU를 놀게 만듦
- **OOM이 날 때까지 늘리는 것이 정답** (이론)
- **실제로는 시도하지 못함 - 시간 부족**

3. Gradient Checkpointing 비활성화 (⭐⭐⭐) - 1.5~2배 속도 향상 ⚠️ AI 소스 기반
4. 데이터타입 통일 (⭐⭐) - 캐스팅 오버헤드 제거 ⚠️ 이론
5. Scheduler 수정 (⭐) - 스케줄링 정확성 ⚠️ 이론
